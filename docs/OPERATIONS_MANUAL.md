## 1. 产品概述

SunGolden AI 工作台是一款面向"需求先行"编程范式的桌面 AI Agent 工具。它将需求澄清、Design 设计、Code 编码和 Write 写作串成完整闭环，帮助用户把 AI Agent 真正放入日常工作中。

### 核心特色

| 特色 | 说明 |
|------|------|
| **需求先行工作流** | 从需求文档 → 设计稿 → 实施计划 → Agent 编码 → 变更审查，完整闭环 |
| **三大工作模式** | Code（代码）、Design（设计）、Write（写作），共用同一运行时 |
| **高性价比模型组合** | 默认集成 DeepSeek、LiteLLM，支持自定义 Provider |
| **本地运行时** | 内建 `kun serve` 运行时，数据默认保存在本机 |
| **权限与安全** | 6 级工具权限模式、文件系统审批、内联 diff 审查 |
| **可视化工作流** | 节点式编排画布，可画、可跑、可复用的 Agent 工作流 |
| **多模态能力** | 图片附件、视觉输入、语音转写、图片/语音/音乐/视频生成 |
| **远程入口** | 支持飞书/微信等 IM 连接、Webhook、定时任务 |

### 产品结构对比

| 模式 | 主要任务 | 产出 |
|------|---------|------|
| **Code** | 对真实代码库对话、编辑文件、运行命令、审查变更 | 代码 diff、计划、Todo、审查结果 |
| **Design** | 从需求生成 UI 设计稿、交互原型、设计流程图 | HTML 原型、设计文档、DESIGN_SYSTEM.md |
| **Write** | 草拟、编辑、润色、导出 Markdown 文档 | Markdown 工作区，支持多种格式导出 |

## 2. 快速入门

### 2.1 环境要求

| 依赖 | 版本 |
|------|------|
| 操作系统 | Windows 10+ / macOS 11+ / Linux |
| 模型服务凭据 | DeepSeek / LiteLLM / 自定义 Provider 至少一个 |

### 2.2 下载与安装

**方式一：下载发布版**

前往内部更新服务器或 GitHub Releases 下载最新版本：

| 平台 | 安装包 |
|------|--------|
| Windows | `.exe`（NSIS 安装器，x64） |
| macOS | `.dmg` 或 `.zip`（Intel / Apple Silicon） |
| Linux | `.AppImage`（x64） |

**方式二：从源码运行**

```bash
git clone https://github.com/KunAgent/Kun.git
cd Kun
npm install
npm run dev
```

中国大陆使用 npm 镜像：

```bash
npm install --registry=https://registry.npmmirror.com
```

### 2.3 首次启动流程

1. **选择界面语言**（默认简体中文）
2. **选择办公地点**（金丰和/生地楼），自动填充 LiteLLM 网关地址
3. **填写 API Key**（LiteLLM 和 DeepSeek 均需填写）
4. **选择权限模式**（默认 bypass 模式）
5. **进入 Code 模式**，绑定本地项目开始使用

|------|------|------------------|

- 网关地址为只读，由地点联动
- 如需自定义地址，可在设置中的 Provider 页面修改

### 3.3 模型供应商配置

初始化设置中包含两个供应商：

**LiteLLM**（默认供应商）

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| Base URL | LiteLLM 网关地址 | 根据办公地点自动填充 |
| 端点格式 | API 协议格式 | `messages`（Anthropic 格式） |
| API Key | 认证密钥 | 必填 |
| 可用模型 | 从 API 拉取 | 点击按钮自动获取 |

> **获取 API Key**：如果没有 key，请联系高翔获取。

**DeepSeek**

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| Base URL | API 地址 | `https://api.deepseek.com` |
| 端点格式 | API 协议格式 | `chat_completions`（OpenAI 兼容） |
| API Key | 认证密钥 | 必填 |

### 3.4 权限模式

系统提供 6 种权限模式，控制 AI Agent 执行操作的审批级别：

| 权限模式 | 说明 | 适用场景 |
|----------|------|---------|
| **始终询问** | 每次工具使用都需要人工审批 | 最高安全级别，适合敏感项目 |
| **只读** | 仅允许读取操作，禁止写入和命令执行 | 代码审查、学习参考 |
| **敏感操作询问** | 仅敏感操作需要审批，常规自动放行 | 平衡安全与效率 |
| **工作区写入** | 工作区内的写入和命令自动放行 | 日常开发推荐 |
| **受信任工作区** | 工作区内操作自动放行，更宽松 | 信任的项目环境 |
| **绕过（Bypass）** | 完全自动执行，无需审批（默认） | 信任的私人环境 |

### 3.5 主题设置

- **跟随系统**（默认）：自动匹配系统主题
- **浅色模式**
- **深色模式**

## 4. 核心工作模式

### 4.1 Code 代码模式

Code 模式是面向真实代码库开发的 AI 编程工作台。

#### 4.1.1 工作区绑定

- 点击左侧边栏可设置/切换工作区目录
- AI Agent 将在该目录下读取文件、搜索代码、执行命令
- 支持多工作区切换

#### 4.1.2 对话交互

**发起对话**：
1. 在底部输入框（Composer）输入问题或指令
2. 支持 `@路径` 引用特定文件/目录
3. 拖拽文件到输入框自动生成 `@路径` 引用
4. 可以附加图片作为视觉输入

**模型选择**：
- 在输入框左侧可选择当前对话使用的模型
- 模型列表由设置中的 Provider 配置决定

**推理力度**：
- 可选择 AI 的推理深度（低/中/高/极高/最大）
- 复杂任务建议提高推理力度

**工具调用**：
- AI 可以读写文件、搜索代码、执行终端命令
- 每次工具调用可在权限模式下审批
- 变更前展示内联 diff，支持逐项审查

#### 4.1.3 需求驱动开发

Code 模式支持完整的需求驱动工作流：

1. **新建需求草稿**：在 SDD 面板创建需求文档
2. **AI 澄清需求**：让 AI 帮助完善和结构化需求
3. **生成实施计划**：使用 `/plan` 命令生成结构化计划
4. **Todo 跟踪**：计划步骤转为 Todo 列表
5. **Agent 编码**：AI 按 Todo 逐步执行
6. **变更审查**：使用 `/review` 对变更进行审查

#### 4.1.4 会话管理

- **新建会话**：创建新的对话线程
- **分叉**：从当前会话分叉出新分支
- **归档**：完成的会话可归档保存
- **搜索**：支持搜索历史会话
- **旁支对话**：在主导航外开启临时讨论

#### 4.1.5 上下文管理

- 拖拽文件只生成 `@路径` 引用，不自动注入全文
- AI 使用 read 工具按需读取文件
- 单文件上下文上限：20,000 字符
- 总计上下文上限：60,000 字符
- 会话过长自动触发上下文压缩

#### 4.1.6 终端集成

- 内嵌 xterm 终端
- 支持 AI 执行 Shell 命令
- 命令执行结果实时展示
- 支持终端字体大小设置

### 4.2 Design 设计模式

Design 模式是 AI 驱动的 UI 设计工作站。

#### 4.2.1 设计循环

1. **描述设计需求**：在右侧面板输入设计描述
2. **配置设计上下文**：
   - 品牌色（颜色选择器）
   - 设计语气（专业/活泼/简约等）
   - 设计系统预设（14 种可选）
   - 结构化设计令牌（圆角、密度、字体）
3. **AI 生成 HTML 原型**：AI 生成独立的 HTML 文档到 `.kun-design/` 目录
4. **画布实时预览**：中间画布实时渲染生成结果
5. **迭代修改**：在当前版本基础上继续修改，自动保存各版本

#### 4.2.2 设计系统预设

共 14 种预设可选：

| 预设 | 风格 |
|------|------|
| shadcn | 现代简约 |
| radix | 无障碍优先 |
| material | Material Design |
| iOS | Apple 风格 |
| fluent | Microsoft Fluent |
| ant | Ant Design |
| chakra | Chakra UI |
| carbon | IBM Carbon |
| polaris | Shopify Polaris |
| bootstrap | Bootstrap |
| geist | Vercel Geist |
| brutalism | 粗野主义 |
| editorial | 编辑风格 |
| none | 无预设 |

#### 4.2.3 画布视图

- **Preview（预览）**：渲染 HTML 原型，支持交互
- **Code（代码）**：Syntax-highlighted 代码视图（shiki）
- **Live（实况）**：展示 Code 模式中运行的真实应用

#### 4.2.4 视口与设备模拟

| 视口 | 宽度 | 用途 |
|------|------|------|
| **手机** | 375px | 移动端设计 |
| **平板** | 768px | 平板端设计 |
| **桌面** | 1280px+ | 桌面端设计 |

支持设备框显示、浅色/深色背景切换。

#### 4.2.5 Design ↔ Code 集成

这是 Design 模式的核心优势：

1. **实现到代码**（"Implement in code"）：
   - 发布共享设计系统 `DESIGN_SYSTEM.md`
   - 构建实现提示
   - 自动打开新的 Code 线程开始编码
   - 记录设计实现关系

2. **反向设计**（"Redesign"）：
   - 从现有 UI 文件逆向生成可迭代的 HTML 模型
   - 从 SDD 需求启动设计流程

3. **设计漂移检测**：
   - ⟳ 标记：设计更新后未重新实现
   - ⚠ 标记：设计系统与原实现不同步
   - ✓ 标记：设计与实现同步

#### 4.2.6 节点画布（Graph Artifacts）

图形化设计流程编排：

- **Prompt 节点**：承载文本/上下文
- **Design 节点**：生成 HTML 设计稿
- **Image 节点**：生成图片（多模态）
- **拓扑执行**：按依赖关系顺序执行
- **实时状态**：每个节点的运行/完成/错误状态

#### 4.2.7 导出

| 导出格式 | 说明 |
|----------|------|
| HTML | 独立 HTML 文件，包含完整样式和脚本 |
| PDF | 通过隐藏浏览器窗口渲染导出 |

#### 4.2.8 反向设计与创作基线

- 内建反 AI-垃圾（anti-AI-slop）规则：
  - 避免奶油色/沙色背景
  - 避免紫色→蓝色渐变
  - 避免弹跳缓动
  - 避免嵌套卡片
  - 避免低对比度灰色文本
  - 提供 `prefers-reduced-motion` 无障碍支持
  - 使用真实字体级别和统一间距尺度

### 4.3 Write 写作模式

Write 模式是独立的 Markdown 写作工作区。

#### 4.3.1 文件管理

- **文件树**：左侧显示工作区内的 Markdown 文件

#### 4.3.2 编辑器视图

| 视图模式 | 说明 |

| **自动保存** | 可配置自动保存间隔 |

#### 4.3.4 字体排版设置

| 字体预设 | 说明 |
|----------|------|
| 系统默认 | 使用操作系统默认字体 |

#### 4.3.5 导出格式

#### 4.3.6 AI 写作预设

- 可配置多个 AI 写作 Agent 预设

#### 4.3.7 写作辅助

- **信息图**：可生成可视化信息图内容
- **原型文档**：可从写作生成交互式原型

| 供应商 | API 格式 | 默认模型 | 说明 |
|--------|---------|---------|------|
| **LiteLLM**（默认） | Anthropic Messages | 从网关动态拉取 | SunGolden 内部网关 |
| **DeepSeek** | OpenAI Chat Completions | deepseek-v4-pro, deepseek-v4-flash | 文本与推理主模型 |
| **Xiaomi MiMo** | OpenAI Chat Completions | mimo-v2-omni 等 | 多模态与语音 |
| **MiniMax** | Anthropic Messages | MiniMax-M2.5 等 | 完整媒体生成 |
| **Vercel AI Gateway** | OpenAI Chat Completions | 从网关拉取 | 多供应商网关 |
| **Zhipu Coding Plan** | OpenAI Chat Completions | glm-5.2 等 | 编程订阅 |

#### 5.2.2 供应商分类

- **套餐订阅**：Token Plan 套餐或订阅制服务

#### 5.2.3 供应商配置项

| 配置项 | 说明 |

## 8. 工作流（创建 Loop）

### 8.1 概述

「创建 Loop」是可视化工作流编排系统，将多步 Agent 流程画成可运行、可复用的工作流。

### 8.2 工作流节点类型

| 节点类型 | 功能 |
|---------|------|
| **AI 节点** | 调用 AI 模型执行任务 |
| **条件节点** | 根据条件分支 |
| **HTTP 请求节点** | 发送 HTTP 请求 |
| **代码节点** | 执行代码片段 |
| **Loop 节点** | 循环执行子工作流 |
| **子工作流节点** | 调用另一个工作流（执行一次） |

### 8.3 Loop 节点详解

#### 8.3.1 两种循环模式

**条件循环（Condition / loop-agent）**

意图：一直干到目标满足。

- 每轮输出折回下一轮输入
- 停止条件达标则停
- `maxIterations` 兜底硬顶
- 输出含 `_iterations`（实际轮数）和 `_done`（是否达标）

**遍历数组（For Each）**

意图：对列表每个元素各跑一遍循环体。

- 顺序执行或并行执行
- 并行保证输出顺序与输入一致
- `continueOnError`：某元素出错后继续/停止

#### 8.3.2 配置项

| 配置项 | 说明 | 范围 |
|--------|------|------|
| `workflowId` | 循环体工作流 ID | - |
| `mode` | `condition` / `foreach` | - |
| `maxIterations` | 最大迭代次数 | 1-100，默认 10 |
| `leftExpr/operator/rightValue` | 停止条件表达式 | condition 模式 |
| `arraySource` | 数组来源表达式 | foreach 模式 |
| `execution` | `sequential` / `parallel` | foreach 模式 |
| `concurrency` | 并行并发数 | 1-8，默认 4 |
| `continueOnError` | 出错策略 | foreach 模式 |

#### 8.3.3 循环变量

循环体内任意节点可使用 `{{ }}` 插值：

| 变量 | 含义 |
|------|------|
| `{{$loop.index}}` | 当前轮/当前元素下标（从 0 开始） |
| `{{$loop.item}}` | 当前元素（foreach）或上轮输出（condition） |
| `{{$loop.total}}` | 总数 |

#### 8.3.4 安全护栏

| 护栏 | 限制 |
|------|------|
| 单 Loop 最大轮数 | 100 |
| 嵌套深度 | ≤ 5 层 |
| 单次运行最大节点数 | 200 |
| 总运行时长限制 | 有超时保护 |

#### 8.3.5 钩子触发

工作流可以绑定到 Code 模式 Agent 的钩子上：
- 写完代码自动 Review
- 保存文件自动格式化
- 提交前自动检查

防递归机制：触发器的 Loop 正在跑时，不会再次触发。

#### 9.1.1 MCP 配置文件

配置路径：`~/.kun/mcp.json`

在 Agent 设置页的「MCP」部分进行管理：
- **查看配置**：加载并显示当前 MCP 配置
- **编辑配置**：直接编辑 JSON 配置文本
- **保存配置**：保存并生效
- **打开目录**：在文件管理器中打开配置目录

#### 9.1.2 MCP 服务器配置

```json
{
  "mcpServers": {
    "server-name": {
      "command": "node",
      "args": ["path/to/server.js"],
      "env": { "KEY": "value" },
      "enabled": true,
      "trustScope": "user"
    }
  }
}
```

| 字段 | 说明 |
|------|------|
| `command` | 启动命令 |
| `args` | 命令参数 |
| `env` | 环境变量 |
| `enabled` | 是否启用 |
| `trustScope` | 信任范围：`user`（全局）/ `workspace`（工作区） |
| `transport` | 传输方式：`stdio` / `sse` |
| `workspaceRoots` | 限制的工作区目录 |

#### 9.1.3 MCP 权限摘要

- 显示已启用的服务器数量
- 显示全局/工作区范围分类
- 显示本地/远程传输类型
- 显示配置了环境变量的服务器
- 检测配置语法错误

### 9.2 Skills

Skills 是可复用的 Agent 行为定义，告诉 AI 如何完成特定的工作。

#### 9.2.1 Skill 来源

| 来源 | 路径 | 说明 |
|------|------|------|
| **全局 Skills** | `~/.kun/skills/` | 对所有工作区生效 |
| **项目 Skills** | `<project>/.kun/skills/` | 仅对当前项目生效 |
| **插件 Skills** | `~/.kun/plugins/*/skills/` | 由安装的插件提供 |
| **内建 Skills** | 系统自动 Seed | 如 design-system、generate-pptx、generate-docx |

#### 9.2.2 内建 Skills

| Skill | 功能 | 触发词 |
|-------|------|--------|
| **design-system** | 设计系统与反 AI 垃圾指南 | 设计相关提示 |
| **generate-pptx** | 生成 PPT 演示文稿 | 生成PPT / 做成PPT / create presentation |
| **generate-docx** | 生成 Word 文档 | 生成Word / 写报告 / 会议纪要 |

#### 9.2.3 Skills 启用/禁用

- 在 Agent 设置页面查看所有 Skills 根目录
- 按目录级别启用/禁用
- 禁用后该目录下所有 Skills 不可用

#### 9.2.4 Skills 模板

每个 Skill 是一个包含 `SKILL.md` 的目录：

```
my-skill/
  SKILL.md        # Skill 定义（名称、描述、触发条件、指令）
  assets/         # 可选：辅助资源
  scripts/        # 可选：辅助脚本
```

## 10. 插件系统

### 10.1 插件仓库

插件可以从以下来源安装：
- **Git 仓库**：GitHub / Gitee 上的插件仓库
- **npm 包**：npm 发布的插件包
- **本地路径**：本地开发的插件

### 10.2 插件安装

**从 Marketplace 安装**：
1. 打开插件市场
2. 搜索或浏览可用插件
3. 点击安装
4. 系统自动检测 marketplace.json，安装真正的插件子目录

**从 Git 仓库安装**：
1. 在插件管理页面输入 Git 仓库 URL
2. 系统自动检测仓库类型：
   - Marketplace 仓库（含 `.claude-plugin/marketplace.json`）
   - 单插件仓库（含 `.claude-plugin/plugin.json`）
   - 兼容模式（有 commands/skills 原生结构）

### 10.3 插件格式兼容

兼容 Claude Code 插件格式：

| 字段 | 兼容处理 |
|------|---------|
| 标识符 | 优先 `name`，fallback `id` |
| 展示名 | 优先 `displayName`，fallback `name` |
| 作者 | 兼容 `string` 和 `{name, email?}` 两种格式 |
| 关键词 | `keywords` / `skills` / `commands` |

### 10.4 文件白名单

安装时自动复制的文件类型：
- 插件元数据：`plugin.json`, `SKILL.md`
- 资源文件：`.shp`, `.shx`, `.dbf`, `.prj`（GIS 矢量格式）
- 栅格数据：`.tiff`, `.tif`
- 脚本：`.py`, `.sh`, `.bat`
- 代理定义：`agents/` 目录

> 取消单文件 5MB 限制，GIS 大文件可正常安装。

### 10.5 插件更新

| 来源类型 | 更新方式 |
|---------|---------|
| `npm:xxx` | npm 更新 |
| `https://github.com/...` | Git Pull |
| `https://gitee.com/...` | Git Pull |
| 本地路径 | 提示用户手动更新 |

### 11.2 文档导出（Write 模式）

|----------|------|
| HTML | 完整的网页文档 |
| PDF | 通过浏览器渲染导出 |
| DOC | Word 97-2003 格式 |
| DOCX | Word 新版格式 |

### 11.3 PPTX 生成

使用内建 skill `generate-pptx`：

**4 套中文模板**：

| 模板 | 配色 | 适用场景 |
|------|------|---------|

| **warm** | 暖橙色 | 商务演示/企业介绍 |

**触发方式**：

## 14. 配置与数据存储

### 14.1 数据存储位置

| 数据 | 路径 |
|------|------|
| GUI 设置 | Windows: `%APPDATA%/Kun/kun-settings.json` |
|          | macOS: `~/Library/Application Support/Kun/kun-settings.json` |
|          | Linux: `~/.config/Kun/kun-settings.json` |
| 运行时配置 | `~/.kun/data/config.json` |
| 会话数据 | `~/.kun/data/` |
| MCP 配置 | `~/.kun/mcp.json` |
| Skills | `~/.kun/skills/` |
| 插件 | `~/.kun/plugins/` |
| 日志 | 可在设置中打开日志目录 |
| 设计工件 | `<workspace>/.kun-design/` |
| SDD 需求 | `<workspace>/.kunsdd/` |

### 14.2 启动配置读取顺序

1. GUI 读取 `kun-settings.json`
2. GUI 同步 `<dataDir>/config.json`
3. Kun serve 读取 `<dataDir>/config.json`
4. CLI 参数和环境变量覆盖基础字段
5. AgentLoop 加载模型配置和压缩阈值

### 14.3 环境变量

| 变量 | 用途 |
|------|------|
| `R2_PUBLIC_BASE_URL` | 覆盖更新基础 URL |
| `R2_RELEASE_PREFIX` | 覆盖更新路径前缀 |
| `KUN_MODEL` | 覆盖默认模型 |
| `KUN_PORT` | 覆盖运行时端口 |

### 14.4 数据隐私

- 所有会话、日志、偏好设置默认保存在本机
- 模型调用通过用户自己的 API Key 发起
- 文件读写和命令执行经权限控制

## 15. 常见问题

### 15.1 如何获取 API Key？

- **LiteLLM**：联系高翔获取
- **DeepSeek**：在 deepseek.com 注册获取
- **其他供应商**：在各供应商官网注册获取

### 15.2 模型无法连接？

1. 检查 API Key 是否正确
2. 检查 Base URL 是否可访问
3. 检查网络代理设置
4. 查看 LLM 调试日志排查具体错误

### 15.3 如何切换工作区？

- Code 模式：在左侧边栏点击工作区选择器
- Write 模式：在设置中配置工作区路径
- Design 模式：设计工件保存在 `.kun-design/` 目录

### 15.4 会话上下文过长怎么办？

- 系统自动触发上下文压缩
- 上下文阈值由模型 profile 配置决定
- 可在 `config.json` 中自定义压缩阈值

### 15.5 如何安装插件？

1. 打开插件市场或输入插件仓库 URL
2. 支持 GitHub/Gitee 仓库、npm 包
3. 系统自动检测并安装
4. 安装后 skills 立即可用，无需重启

### 15.6 如何备份数据？

备份以下目录：
- `~/.kun/`（运行时数据、配置、Skills、插件）
- `%APPDATA%/Kun/`（GUI 设置，Windows）
- 工作区内的 `.kun-design/`、`.kunsdd/`

### 15.7 权限模式如何选择？

| 场景 | 推荐模式 |
|------|---------|
| 生产环境操作 | 始终询问 |
| 代码审查/学习 | 只读 |
| 团队协作 | 敏感操作询问 |
| 日常开发 | 工作区写入 |
| 个人项目 | 受信任工作区 / 绕过 |

### 15.8 如何更新软件？

1. 自动检查更新（启动时）
2. 手动：设置 → Agent → 检查更新
3. 下载并安装更新
4. 或从内部服务器下载新版本手动安装
